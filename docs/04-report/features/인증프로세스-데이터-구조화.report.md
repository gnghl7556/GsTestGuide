## 7. Lessons Learned & Retrospective

### 7.1 What Went Well (Keep)

1. **Design-First Approach**
   - Comprehensive design document (9 sections, 493 lines) provided clear implementation roadmap
   - Phase-by-phase structure prevented scope creep
   - Frontmatter schema design avoided implementation rework

2. **Build-Time Parsing Strategy**
   - Virtual modules eliminated runtime performance overhead
   - HMR support improved development experience
   - Type safety maintained through `.d.ts` declarations

3. **Minimal Dependencies**
   - Only `gray-matter` added (< 100KB)
   - No complex markdown parser dependencies (remark/unified)
   - Regex-based parsing kept bundle size small

4. **Obsidian Compatibility**
   - Pure markdown + YAML frontmatter works seamlessly
   - No custom syntax ([[links]] avoided)
   - 7-stage folder structure intuitive for navigation

### 7.2 What Needs Improvement (Problem)

1. **Code Duplication Issue**
   - Vite plugin contains inline markdown parsing logic (L27-79)
   - Duplicates functionality in `src/lib/content/markdownUtils.ts`
   - **Impact**: Must update parsers in 2 places when logic changes
   - **Root Cause**: Build-time execution cannot import runtime modules

2. **Incomplete Cleanup**
   - Old data files (`src/data/requirements.ts`, etc.) still exist
   - **Risk**: Developers might accidentally import from old files
   - **Cause**: Forgot Phase 4 Step 4-3 during implementation rush

3. **Missing 1 Process File**
   - Design expected 16 process files, only 15 created
   - **Uncertainty**: Either design count is incorrect or 1 file missed
   - **Action**: Verify design document intent

4. **Undocumented Optional Fields**
   - "ì°¸ì¡° ì •ë³´" and "ë‹´ë‹¹ì" table sections not in all process files
   - No content writing guide for future contributors
   - **Risk**: Schema inconsistency over time

### 7.3 What to Try Next (Try)

1. **Extract Shared Parser Module**
   - Create `lib/markdown-parser-node.js` (Node.js module)
   - Import in both Vite plugin and runtime parsers
   - Reduce duplication from ~150 LOC to single source

2. **Add Parser Unit Tests**
   - Test `parseCheckboxList()`, `parseTable()`, etc.
   - Mock markdown inputs and verify output
   - Prevent regressions when refactoring

3. **Create Content Writing Guide**
   - Document required vs optional H2 sections
   - Provide frontmatter examples
   - Define table format standards

4. **Add JSON Schema Validation**
